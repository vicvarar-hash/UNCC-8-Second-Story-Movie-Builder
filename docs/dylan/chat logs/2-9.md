## L6: Pipeline-Aware Design and Gate Planning

L6 shifts from “what the system should look like” (L5) to **whether that design can be evaluated continuously** in a pipeline. The central idea: **design that can’t be evaluated through the pipeline is incomplete.**

---

## Steps for L6 (from “What You Will Do”)

1. **Revisit Design v1**  
   Start from the design you produced in L5.

2. **Refine design with pipeline constraints in mind**  
   - Design for **observability**, not only functionality.  
   - Ask: How will we observe behavior? Detect failure? Collect evidence?  
   - Make sure the pipeline can “see” what matters; if it can’t see it, it can’t evaluate it.

3. **Identify AI entry points and trust boundaries**  
   - Where does AI generate artifacts?  
   - Where does AI influence decisions?  
   - Where do AI explanations appear?  
   - These must be explicit so the pipeline can evaluate trust later.

4. **Propose initial trust gates (even placeholders)**  
   - Derive gates from: trust hypotheses, design risks, and AI-generated or AI-influenced artifacts.  
   - Gates can be weak, naive, or incomplete at this stage.  
   - A weak gate that runs is better than a perfect gate that doesn’t exist.

5. **Map requirements → design → pipeline stages**  
   Produce an explicit mapping:  
   - **Requirements → components**  
   - **Components → pipeline stages**  
   - **Pipeline stages → candidate trust gates**  
   This mapping is the conceptual blueprint for later DevOps and evaluation work.

---

## Expected outcomes (by end of L6)

- **Design v1** (or v1.1) is committed to the repo and is explicitly **pipeline-aware**.  
- **AI usage points** are visible in the design.  
- **Candidate trust gates** are identified (even as placeholders).  
- A **requirements → components → trust gates** mapping exists.  
- Correctness is not required; **executability and observability** are.

---

## Red flags to avoid

- Assuming the pipeline will “handle it later.”  
- AI logic that is either embedded everywhere or nowhere (no clear boundaries).  
- Postponing trust gates “until we understand more.”  
- Designs where failures cannot be detected by the pipeline (such designs need to be revised).
