### Not yet in the suite (07, 08, 09)

These contracts exist but are commented out in `run-contracts.mjs`. Against the current app, they would fail:

| Contract | Name | Status | What’s missing |
|----------|------|--------|----------------|
| **07** | Safe JSON Parse with Error Surfacing | Would fail | `gemini.ts` has bare `return JSON.parse(response.text \|\| '{}')` (lines 173, 331) and `JSON.parse(response.text \|\| '{"shots":[]}')` (line 279) without try/catch. Parse errors can surface as empty objects instead of explicit errors. |
| **08** | Video Generation Timeout (TR7) | Would fail | Polling loop has `while (!operation.done)` but no timeout (no `TimeoutError`, `maxWait`, `deadline`, etc.). The loop can run indefinitely. |
| **09** | Plan Structure Validation | Would fail | No `validatePlan` / `validateShotPlan` (or similar) and no explicit checks for required fields (`narrativeIntent`, `videoPrompt`, `action`) before creating shots. |

---

## Suggested order to implement

1. **Contract 07 (Safe JSON Parse)** – Smallest change, high impact  
   - Wrap all `JSON.parse` calls in try/catch  
   - Throw with a clear message on parse failure instead of returning `{}`  
   - Directly addresses invalid rubrics/plans becoming empty objects with no visibility  

2. **Contract 09 (Plan Structure Validation)** – Medium effort  
   - Add validation before creating shots from parsed plans  
   - Check required fields (`narrativeIntent`, `videoPrompt`, `action`) and throw/reject when missing  
   - Helps with schema drift (H-S2)  

3. **Contract 08 (Video Timeout)** – More involved  
   - Add a max wait (e.g. 5 minutes) around the Veo polling loop  
   - Throw `TimeoutError` when exceeded  
   - Prevents indefinite hangs (TR7)
