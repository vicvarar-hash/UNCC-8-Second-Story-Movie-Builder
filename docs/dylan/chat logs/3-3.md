**Cursor**


**vite.config.ts** – Used by Vite for dev, build, and preview. It configures things like:
- Dev server (port, host)
- Plugins (e.g. React)
- Env variables (`loadEnv`, `define`)
- Path aliases (`resolve.alias`)

**vitest.config.ts** – Used by Vitest when running tests. It:
- Uses the same config format as Vite (Vitest is built on Vite)
- Adds a `test` block with Vitest-specific options (`environment`, `globals`, etc.)
- Is loaded instead of `vite.config.ts` when Vitest runs (Vitest prefers `vitest.config.*`)

**Important:** With a separate `vitest.config.ts`, Vitest does not use your `vite.config.ts`. So your Vitest setup does not get:
- `loadEnv` / `define` for `GEMINI_API_KEY`
- The `@` path alias

**Simpler approach:** Put the Vitest config into `vite.config.ts` and remove `vitest.config.ts`:

```ts
// vite.config.ts - add a test block
export default defineConfig(({ mode }) => {
  // ... existing config
  return {
    // ... existing
    test: {
      environment: "node",
      globals: true,
    },
  };
});
```

Then Vitest will use the same env, aliases, and plugins as the rest of the app.

**`tests/apiKeyValidation.test.ts`** — 3 tests for `validateApiKeyLogic`:

1. **Throws when key is empty** — `validateApiKeyLogic("")` throws with the expected API key error message.
2. **Throws when key is too short (< 10 chars)** — `validateApiKeyLogic("short")` and `validateApiKeyLogic("123456789")` both throw.
3. **Does not throw when key is valid (>= 10 chars)** — `validateApiKeyLogic("1234567890")` and `validateApiKeyLogic("a".repeat(20))` do not throw.

`validateApiKeyLogic` is a pure helper in `services/gemini.ts` that checks `!key || key.length < 10` and throws. The main `validateApiKey()` calls it after `getCurrentApiKey()`, so the tests focus on the validation logic without browser mocks.
