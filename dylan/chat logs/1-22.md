## Critical Assessment: System Requirements vs. Codebase Reality

### Major gaps and trust issues

#### 1. Missing requirement: evaluation mode and outcome coding
- Code reality: The system has `evaluationMode: true` and an `Outcome` interface with fields: `failureType`, `failureVisibility`, `trustCalibration`, `correctionEffort`, and `notes`.
- Requirements gap: No requirement mentions evaluation mode, outcome coding, or structured failure classification.
- Trust issue: The requirements don't acknowledge that the system is built for research/evaluation, not just production. This is a critical assumption that should be explicit.

#### 2. Missing requirement: AI self-review with prompt revision
- Code reality: `runAISelfReview()` generates a rubric with scores, reasoning, and optional `promptRevision` with a revised prompt and change list.
- Requirements gap: FR5 mentions "evaluative feedback" but not structured self-review, rubric scoring, or prompt revision suggestions.
- Trust issue: The system uses AI to review AI outputs and suggest fixes. This recursive AI usage should be explicit in requirements, with risks documented.

#### 3. Missing requirement: attempt lineage and regeneration tracking
- Code reality: `GenerationAttempt` includes `promptSource` ('original' | 'self_review_revision' | 'manual_edit'), `parentAttemptId`, and a full attempts array per shot.
- Requirements gap: No requirement mentions tracking multiple attempts, lineage, or regeneration history.
- Trust issue: The requirements don't capture that users can regenerate with different sources, which affects trust evaluation and evidence collection.

#### 4. Missing requirement: validation phase for external videos
- Code reality: A "validate" phase allows uploading external videos (clips or full movies) for AI review, separate from the production workflow.
- Requirements gap: No requirement mentions validating external videos or a separate validation workflow.
- Trust issue: This feature suggests the system is used to evaluate videos from other sources, which should be a requirement.

#### 5. Missing requirement: evidence bundle export
- Code reality: `exportBundle()` exports the entire project as JSON, including plans, attempts, lineage, outcomes, and self-reviews.
- Requirements gap: FR6 mentions "export" but only for assembled videos, not evidence bundles.
- Trust issue: The system is designed to export evidence for research/evaluation, which should be explicit.

#### 6. Hardcoded assumptions not documented
- Code reality:
  - `useSeed: true, useRefImage: false, requestExplanation: true` are hardcoded in `handleGenerateShot()` (line 178)
  - Video generation always uses `veo-3.1-generate-preview` with fixed `720p` and `16:9`
  - Self-review always uses `gemini-3-flash-preview`
  - Shot plans always use `gemini-3-pro-preview`
- Requirements gap: No requirement specifies model choices, resolution constraints, or default generation parameters.
- Trust issue: These are implementation decisions that affect trust evaluation but aren't captured in requirements.

#### 7. Continuity mechanism implementation gap
- Code reality: `generateVideoAttempt()` accepts `options.useSeed` and `previousVideoUrl`, but the code doesn't actually pass the previous video's final frame as a seed.
- Requirements gap: FR4 says "may optionally provide... the final frame of the previous shot" but doesn't specify when or how.
- Trust issue: The requirement suggests continuity support, but the implementation may not fully realize it, creating a trust gap.

#### 8. Missing requirement: preset scripts for trust testing
- Code reality: Three preset scripts are hardcoded: "Continuity Drift", "Plausible Planning Errors", "Misleading Explanations".
- Requirements gap: No requirement mentions test scenarios or preset scripts.
- Trust issue: The system is designed with specific failure modes in mind, which should be documented as requirements or assumptions.

#### 9. LocalStorage persistence assumption
- Code reality: Projects are saved to `localStorage` with key `"EVAL_PROJECT"`.
- Requirements gap: Assumptions mention "best-effort" persistence but don't specify localStorage or session-only storage.
- Trust issue: Evidence may be lost if localStorage is cleared, which should be explicit.

#### 10. Missing requirement: structured shot plan schema
- Code reality: `ShotPlan` has 25+ fields (narrativeIntent, shotType, cameraMovement, lens, continuityLocks, mustInclude, etc.).
- Requirements gap: FR2 mentions "observable fields such as action, camera behavior, mood or lighting" but doesn't specify the full schema.
- Trust issue: The requirements don't capture the complexity of the shot plan structure, which affects how AI interprets and generates plans.

### Trust-critical requirements not prioritized

The code reveals implicit priorities that should be explicit:

- Critical (system breaks without it):
  - Evidence collection and export (needed for evaluation)
  - Artifact visibility (needed for trust assessment)
  - Human authority over acceptance/rejection

- Important (system functions but trust evaluation is compromised):
  - AI self-review accuracy
  - Continuity mechanisms
  - Attempt lineage tracking

- Desirable (nice to have):
  - Prompt revision suggestions
  - Validation of external videos
  - Preset test scripts

### Recommendations

1. Add explicit requirements for:
   - Evaluation mode and outcome coding
   - AI self-review with structured rubric
   - Attempt lineage and regeneration tracking
   - Evidence bundle export
   - Validation phase for external videos

2. Document hardcoded assumptions:
   - Model choices and versions
   - Default generation parameters
   - Resolution and aspect ratio constraints
   - LocalStorage persistence limitations

3. Prioritize trust-critical requirements:
   - Mark which requirements are critical vs. important vs. desirable
   - Justify prioritization based on evaluation needs

4. Clarify continuity mechanisms:
   - Specify when seeds are used
   - Document how continuity is attempted vs. guaranteed
   - Acknowledge implementation gaps

5. Add discrepancy detection section:
   - Document where AI was used in requirements elicitation
   - Identify where code reveals assumptions not in requirements
   - Note where requirements are more confident than implementation supports

This assessment shows the requirements are incomplete relative to the implementation. The code reveals the system is built for research/evaluation, with features (outcome coding, evidence export, AI self-review) not captured in the requirements. These gaps should be addressed to align requirements with the codebase and course expectations.

## Critical Backend Trust Gap Analysis

### Silent failure patterns

#### 1. JSON parsing with silent fallbacks
**Location:** `services/gemini.ts` lines 134, 217, 245

```typescript
// Line 134: runAISelfReview
return JSON.parse(response.text || '{}');

// Line 217: generateProjectPlan  
const parsed = JSON.parse(response.text || '{"shots":[]}');
return parsed.shots;

// Line 245: suggestNextShotPlan
return JSON.parse(response.text || '{}');
```

**Trust issue:**
- If `response.text` is null/undefined, falls back to empty objects
- If JSON is malformed, `JSON.parse` throws, but there's no try-catch
- If the schema is violated, the system proceeds with invalid data
- No validation that parsed JSON matches the expected structure

**Impact:**
- Empty shot plans could be created silently
- Self-review could return empty rubrics
- Frontend may render invalid data as if it were valid

**Evidence gap:** These failures are not logged or surfaced, making them hard to detect.

### Recommendations for backend trust hardening

#### Critical (fix immediately)
1. Add try-catch around all JSON.parse calls with proper error handling
2. Implement runtime validation of parsed JSON against schemas
3. Add timeout and retry limits to video generation polling
4. Implement continuity seed mechanism or remove the claim
5. Validate API key before making calls

#### Important (fix soon)
6. Add structured error types and classification
7. Validate frame sampling results before self-review
8. Add validation of shot plans before creating shots
9. Surface errors to the UI, not just console

#### Desirable (improve trust evaluation)
10. Log all AI API calls with request/response for evidence
11. Add metrics for failure rates by type
12. Validate AI response schemas at runtime
13. Add retry logic with exponential backoff

### Where to focus analysis

1. `services/gemini.ts` — all AI integration points
   - JSON parsing and validation
   - Error handling and classification
   - API call logging

2. `App.tsx` — data flow validation
   - Shot plan validation before use
   - Error propagation from services
   - State consistency checks

3. Type safety vs. runtime validation
   - TypeScript types don't prevent runtime errors
   - Need runtime validation at boundaries

The frontend has trust mechanisms (outcome coding, evidence export), but the backend has silent failures that undermine them. These gaps should be addressed before trust evaluation can be reliable.
